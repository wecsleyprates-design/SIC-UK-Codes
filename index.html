1. Problem Statement

Purpose

This document explains exactly how Worth's industry classification pipeline works today, what it produces, what it is missing, and provides a concrete experiment and implementation plan to add a reliable uk_sic_code fact for GB (and other non-US) businesses.

Worth classifies businesses using the North American Industry Classification System (NAICS) — a US-centric 6-digit code. For UK businesses, the authoritative system is the UK Standard Industrial Classification (UK SIC), a 5-digit code maintained by Companies House.

The core problem: UK SIC data flows into Worth's pipeline from two vendors (OpenCorporates and Trulioo) but is silently discarded before it reaches any stored or surfaced fact. No uk_sic_code field exists anywhere in the system — not as a resolved fact, not as a database column, not in scoring.

This document explains why, and exactly how to fix it.

The classification_codes fact — stored but invisible

The classification_codes fact (defined in lib/facts/businessDetails/index.ts L323–344) parses all industry codes from OpenCorporates industry_code_uids into a generic object:

Example classification_codes for UK
{
  "gb_sic":  "62020",    // UK SIC is HERE, but not surfaced as its own field
  "us_naics": "541511",
  "eu_nace": "J.62.02"
}

This fact is emitted on the Kafka facts topic but: It is a raw map with no type guarantees; No downstream consumer parses gb_sic; case-service has no migration; Scoring does not reference it.

Raw data shape from OpenCorporates

"gb_sic-62020|us_naics-541511|eu_nace-J.62.02"

The gb_sic-62020 segment is sourced from **Companies House**. When the prefix gb_sic is present, the code is always a **5-digit UK SIC**. This is the most authoritative UK SIC source available.

4. Layer 2: The Fact Engine — How Resolution Works

Weight vs. Confidence — the difference

ConceptWhat It IsRangeSet By
WeightStatic trust multiplier by design0–1.0Hardcoded in sources.ts
ConfidenceDynamic match quality per business0–1.0Computed at runtime by getter

interaction: factWithHighestConfidence rule picks the candidate with highest confidence. If tie (within 0.05), weight breaks the tie.

Full resolution flow for a UK business

1. FactEngine.match() runs all source getters:
   ├── opencorporates.getter() → parses "gb_sic" and "us_naics"
   └── AINaicsEnrichment.getter() → last resort
2. manualOverride checked first → if exists, DONE
3. rule runs: Highest Confidence wins; Weight breaks ties.

9. Layer 6: The Full Experiment & Validation Plan (Details)

Goal

Determine empirically what data is actually available for UK businesses and identify the gaps before implementation.

Experiment 1 — Measure OpenCorporates UK SIC Coverage

Objective: Measure baseline availability of codes in full 16.6M record dataset.

SELECT COUNT(*) AS total, SUM(CASE WHEN industry_code_uids LIKE '%gb_sic-%' THEN 1 ELSE 0 END)
FROM opencorp.companies WHERE jurisdiction = 'gb';

Results: 66.49% Coverage (11,079,157 records). Findings: Strong baseline, but 33% gap at registry level.

Experiment 2 — Validate Trulioo sicCode for UK Businesses

Objective: Determine if Trulioo returns UK SIC (5-digits) or US SIC (4-digits).

Results: Found 4-digit codes (e.g., "7372") labeled "US Standard Industry Code 1972". Findings: Trulioo defaults to US standards. Verdict: REJECTED.

Experiment 3 — Measure the Gap in Managed Businesses

Objective: Measure coverage specifically for curated/scored businesses.

SELECT COUNT(*) AS total, SUM(...) AS has_sic, (COUNT(*) - SUM(...)) / COUNT(*) AS gap
FROM warehouse.oc_companies_latest WHERE jurisdiction = 'gb';

Results: 37.5% Coverage. Gap: 62.5%. Findings: AI Enrichment (Phase 4) is critical to reaching acceptable coverage levels.

Experiment 4 — Verify OpenCorporates Format Accuracy

Objective: Confirm extracted codes are modern 2007 standard.

Findings: Verified frequencies for codes like 8299| and 70229. **Conclusion**: Verified TRUE UK SIC 2007.

Experiment 5 — AI Prediction Accuracy

Objective: Prove AI can handle gaps where registry data is missing or stale.

Findings: AI correctly identified modern 62090 and 96090 for businesses where CH registry was stuck in 1992 standard. AI is a **Cleansing Tool**.
