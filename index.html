Mapping the Current Bottlenecks

Before we implement the solution, we must understand exactly where the data is lost in our current architecture.

1

Registry Ingestion (OpenCorporates)

The system receives official raw data containing: gb_sic-62020|us_naics-541511. The data exists at the entry point.

2

The Silo Filter (The Failure Point)

Existing logic in integration-service specifically pulls only strings containing "naics". The gb_sic segment is discarded immediately.

3

Vendor "Global" Pollution

Vendors like Trulioo/Equifax return US-centric SIC codes (4-digits). Without a UK-specific fact field, this data "pollutes" our UK businesses with US industry standards.

4

Final Persistence Loss

Since case-service lacks a uk_sic_id column, even if data survived the filter, it has nowhere to be stored. The cycle of data loss is complete.

Phase 2: Registry Extraction Model

The following logic will be implemented in integration-service to reclaim the 37.5% of registry data.

integration-service / lib/facts/businessDetails/index.ts
uk_sic_code: [
  {
    source: sources.opencorporates,
    fn: (_, oc: OpenCorporateResponse) => {
      // Loop through registry strings
      for (const uid of oc.industry_code_uids.split("|")) {
        const [prefix, code] = uid.split("-", 2);
        // Look for the 'gb_sic' prefix we validated in Exp 1
        if (prefix === "gb_sic" && /^\d+/.test(code)) {
          return code.padStart(5, "0"); // Normalize to 5 digits
        }
      }
    }
  }
]
