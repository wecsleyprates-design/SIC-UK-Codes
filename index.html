01. Problem Statement

Worth currently classifies businesses using the **North American Industry Classification System (NAICS)** — a US-centric standard. For UK businesses, the only authoritative standard is the **UK SIC (Standard Industrial Classification) 2007**.

"The core problem: UK SIC data flows into our pipeline from vendors like OpenCorporates but is silently discarded by hardcoded NAICS filters. We are functionally 'Industry Blind' for 6 out of 10 UK businesses."

This document is the definitive technical "Bible" explaining why this happens and providing the three-phase roadmap to fix it while establishing a model for every country in Worth's future expansion.

03. The Narrative: Story of the Data Loss

1

The Ingestion Point

A business record from **OpenCorporates** arrives in our service. The payload is rich: gb_sic-62020|us_naics-541511. The ground truth for the UK is physically inside our infrastructure.

2

The Logic Filter (The Bottleneck)

The code in integration-service begins to "read" the data. It is hardcoded to find the string "us_naics". When it sees "gb_sic", it does not recognize the prefix. It ignores it, drops the data from memory, and proceeds with the US code only.

3

The Pollution (Deceptive Vendors)

A secondary vendor like **Trulioo** checks the same business. It returns a 4-digit code: 7372. Because our system lacks a "Native UK Validator", it accepts this as a valid SIC—but 7372 is a **US SIC 1972** code. We have now polluted a UK record with 50-year-old US data.

4

The Final Loss (Persistence)

Even if the code survived the filter, the case-service database simply doesn't have a place to store it. There is no uk_sic_code column. The data evaporates.

05. Fact Engine Logic & Resolvency

The FactEngine resolve candidates using specific rules. For UK SIC, we implement a weight-heavy rule chain.

lib/facts/factEngine.ts
// Resolvency logic for jurisdiction-aware facts
class FactEngine {
  resolveUK_SIC(candidates: Fact[]) {
    return candidates
      .filter(c => c.value.length === 5) // Rejects US pollution
      .sort((a, b) => b.weight - a.weight)[0]; // Registry wins
  }
}

13. Strategic Executive Answers

Q1: Exactly what happens in the flow?

A: Data converges from 7 vendors. We filter for native prefixes, resolve conflicts based on Registry Weight (0.9) vs AI Weight (0.1), and save the results to the Case Service. The AI prompt acts as the "Gap Closer" for the 62.5% missing records.

Q2: How we handle New SIC UK?

A: By inserting a Parallel Gateway. Instead of forcing UK data through a NAICS-only pipe, we build a dedicated uk_sic_code pathway that mirrors NAICS but enforces 5-digit UK 2007 integrity.

Q3: A model for other countries?

A: Precisely. This is the Worth Global Adapter. By replacing the 'gb_sic' prefix with 'de_wz' (Germany) or 'au_anzsic' (Australia), the entire pipeline instantly scales to the next jurisdiction.
